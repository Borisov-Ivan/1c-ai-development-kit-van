# openspec-docs

Генерация проектных документов на основе спецификаций OpenSpec.

## Принцип

Спецификации в `openspec/specs/`, контекст проекта в `openspec/project.md` и при необходимости `openspec/architecture-extension.md` — источники. Документы генерируются без додумывания: если информации нет в источниках — она не попадает в документ.

## Поддерживаемые типы документов

| Тип | Команда | Промпт | Аудитория |
|-----|---------|--------|-----------|
| Функциональные требования | `/docs:functional` | `prompts/functional.md` | Бизнес-заказчик, пользователи |
| Архитектура (для заказчика) | `/docs:architecture` | `prompts/architecture.md` | Заказчик, стейкхолдеры |
| Технический проект | `/docs:technical` | `prompts/technical.md` | Разработчики, техкоманда |

## Алгоритм

1. **Определить тип документа** из запроса пользователя (`functional` | `architecture` | `technical`).

2. **Прочитать контекст проекта:**
   ```
   openspec/project.md
   ```
   Для типов `architecture` и `technical` дополнительно прочитать:
   ```
   openspec/architecture-extension.md
   ```

3. **Прочитать все спецификации:**
   ```
   openspec/specs/*/spec.md
   ```
   Если у спека нет секции Purpose — определять уместность по Requirements и сценариям.

4. **Определить slug** (см. раздел ниже) и **прочитать промпт** из `prompts/<тип>.md`.

5. **Применить промпт** к собранным данным — сгенерировать документ.

6. **Показать результат** пользователю и спросить, нужны ли корректировки.

7. **Записать** в `docs/<slug>/<имя документа>.md` после подтверждения. Имена файлов: `Функциональные требования.md`, `Архитектура.md`, `Технический проект.md`.

## Определение slug

Slug — имя подкаталога в `docs/`, в который складываются документы по одному решению. Формируется из имени продукта в `openspec/project.md` (секция «Продукт» или «Назначение»): привести к kebab-case, латиница. Пример: «ргОблачныеПодписиKonturCryptoAPI» → `oblachnye-podpisi-kontur-cryptoapi` или кратко `kontur-cryptoapi`. Один и тот же slug используется для всех трёх типов документов одного решения. При нескольких решениях в проекте — отдельный подкаталог на каждое.

## Фильтрация спеков по типу документа

- **Функциональные требования:** спеки, в Purpose (или в Requirements, если Purpose нет) которых упоминаются пользователь, администратор, форма, интерфейс, маркировка, настройка, ограничения. Чисто технические спеки не включаются.
- **Архитектура (для заказчика):** спеки user-facing и mixed — для описания потоков и ролей. Технические спеки — только одной фразой при описании потоков (без деталей).
- **Технический проект:** все спеки без исключения.

Определение «внутренний спек» — по Purpose: если в описании нет упоминания пользователя, администратора, формы или интерфейса, спек считается внутренним. Если у спека нет секции Purpose — категорию определять по содержанию Requirements.
