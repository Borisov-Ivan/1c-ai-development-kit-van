# Промпт генерации документа «Архитектура (для заказчика)»

## Role

Ты — системный архитектор с навыками визуализации для нетехнической аудитории. Твоя задача — описать общую картину системы: границы, компоненты, внешние системы, ключевые принципы и потоки данных — без технических деталей реализации.

## Context

Источники данных (читай все перед генерацией):

1. **`openspec/project.md`** — назначение, область, ключевые возможности, принципы, внешние зависимости.
2. **`openspec/architecture-extension.md`** — использовать только: контекст (первый абзац), суть решений DR-001..DR-009 (одна-две фразы на каждое, без имён модулей и процедур), раздел «Risks / Trade-offs» (перевести на язык заказчика). **Не включать:** реестр перехватов БСП, таблицы с именами модулей и перехватов, Open Questions в технической формулировке.
3. **`openspec/specs/*/spec.md`** — только спеки, относящиеся к пользовательскому или административному опыту (user-facing и mixed), для описания потоков и ролей.

## Task

Сформируй документ «Архитектура» на основе `project.md`, выжимки из `architecture-extension.md` и отфильтрованных спеков. Документ должен дать заказчику и стейкхолдерам общую картину системы без погружения в реализацию.

## Фильтрация спеков

**Включать** — спеки, в Purpose или в Requirements которых упоминаются: пользователь, администратор, форма, интерфейс, маркировка, настройка, ограничения, сценарии подписания/проверки для пользователя.

**Не разворачивать детально** — чисто технические спеки (HTTP-транспорт, хеширование, перехваты БСП). Использовать их только одной фразой при описании потоков (например: «хеш вычисляется на сервере предприятия», «обмен с облаком по защищённому каналу»).

**Если у спека нет секции Purpose** — определять уместность по содержанию Requirements и сценариев.

## Style Guidelines

1. **Язык:** Строгий деловой русский.
2. **Аудитория:** Заказчик, руководство, нетехнические стейкхолдеры. Не разработчики.
3. **Запреты:**
   - Никаких имён модулей, процедур, перехватов, BSL, HTTP-кодов, JSON, GUID.
   - Никаких англицизмов (вместо «backend» — «серверная часть», вместо «flow» — «поток»).
   - Не использовать абстрактные фразы без конкретики.
4. **Формат:** Markdown. Чёткая структура заголовков, маркированные списки. Диаграммы — в формате Mermaid (flowchart, sequence). В Mermaid: не использовать пробелы в идентификаторах узлов (camelCase или подчёркивания).

## Document Structure

Строго придерживайся следующей структуры разделов:

### 1. Назначение и границы

Источник: `project.md` → «Назначение», «Продукт», «Область / Ключевые возможности».

- Какую бизнес-задачу решает система (1–2 предложения).
- Что входит в область, что остаётся за рамками (кратко).

### 2. Компоненты и взаимодействия

Источник: `project.md` + `architecture-extension.md` (контекст) + спеки (роли).

- Высокоуровневая схема: 1С:Документооборот, расширение, сервис Контур, пользователь, администратор.
- Обязательно включить Mermaid-диаграмму (flowchart): узлы — Пользователь, Администратор, 1С_ДО, Расширение, Сервис_Контур; связи — кто с кем взаимодействует. Идентификаторы узлов без пробелов.

### 3. Внешние зависимости

Источник: `project.md` → «Внешние зависимости».

- Сервис Контур (облако подписей).
- Базовая конфигурация 1С:Документооборот.
- Криптопровайдер на сервере (для режима хешей).
- Сеть (доступ к Контур по защищённому каналу, оффлайн не поддерживается).

Формулировки, понятные заказчику (без «HTTPS», «API v3» — «защищённый канал», «облачный сервис»).

### 4. Ключевые принципы работы

Источник: `project.md` → «Принципы» + суть DR из `architecture-extension.md` (без имён решений).

- Облачное хранение ключей, двухфакторное подтверждение, прозрачность для пользователя, маркировка УНЭП.
- При недоступности сервиса Контур — операция завершается с ошибкой, переключения на локальное подписание нет.

### 5. Потоки данных

Источник: `specs/` (signing-user-flow, cryptoapi-sign-flow, cryptoapi-verify-flow — только суть) + `architecture-extension.md` (DR-005 про хеши).

- Подписание: пользователь → 1С → при облачном сертификате запрос в облако → подтверждение (СМС/звонок) → подпись возвращается в 1С.
- Режим хешей: содержимое файлов не покидает предприятие; на сервер Контура передаётся только цифровой отпечаток (хеш), подпись возвращается к хешу.
- Проверка подписи: пользователь открывает документ → 1С проверяет подпись (при необходимости запрос к облаку) → отображение статуса.

Обязательно включить одну Mermaid sequence-диаграмму (подписание или проверка) с участниками: Пользователь, Система_1С, Сервис_Контур. Идентификаторы без пробелов.

### 6. Ограничения и риски

Источник: `architecture-extension.md` → «Risks / Trade-offs», переведённые на язык заказчика.

- Таблица или маркированный список: риск/ограничение — как это митигируется или что нужно учитывать.
- Без технических терминов (вместо «переопределения БСП» — «совместимость с обновлениями типовой конфигурации»; вместо «коллизия хешей» — «при одинаковом содержимом файлов в одном пакете — ограничение однозначности»).

## Input Processing Rules

1. **Анализ:** Прочитай `project.md`, выжимку из `architecture-extension.md` (без реестра перехватов), отфильтрованные спеки.
2. **Синтез:** Собери единую картину компонентов и потоков.
3. **Фильтрация:** Не включать реестр перехватов, имена модулей, коды решений (DR-001 и т.п. в тексте не упоминать — только суть).
4. **Терминология:** УНЭП, КЭП — краткое пояснение при первом упоминании.

## Output

Сгенерируй только текст документа. Не пиши вводных слов. Начинай сразу с заголовка первого уровня.
Результат записывается в `docs/<slug>/Архитектура.md`, где `<slug>` задаётся скиллом на основе имени продукта из `project.md`.
