Провести предрелизное ревью расширения 1С. Результат — структурированный отчёт с замечаниями по уровням. После — предложение создать openspec change на устранение.

**Input**: Опционально — название расширения или путь. Если не указано — определить из контекста или спросить.

---

## Шаг 0. Определение scope

Если пользователь указал конкретное расширение (имя папки в `src/*/cfe/`, файлы или change) — использовать.

Иначе — найти доступные расширения:

```bash
# Найти расширения в src/
dir src\*\cfe\ -Directory
```

Использовать **AskUserQuestion tool** для выбора. Показать список расширений (например, `pav_ИсключениеУчастниковПоУсловию`).

Также проверить активные openspec changes:

```bash
openspec list --json
```

Если есть активный change — уточнить, является ли ревью частью его workflow.

**ВАЖНО**: Не начинать ревью до получения чёткого scope.

## Шаг 1. Сбор контекста

### 1.1 Прочитать стандарты проекта

- `.cursor/rules/1c-coding-standards.mdc` — стандарты BSL проекта.
- `.cursor/agents/onec-code-reviewer.md` — категории и правила ревью (в т.ч. отсылка к вендорским стандартам).
- **Вендорские стандарты 1С** (для полноты предрелизной проверки): `.cursor/skills/1c-vendor-standards/SKILL.md` — чеклисты по доменам; навигация — `.cursor/docs/standard/1c-standards-navigator.md`; полный текст по доменам — `.cursor/docs/standard/std-*.md` (метаданные, запросы, транзакции, формы и т.д.). Согласовано с AGENTS.md и onec-code-reviewer.

### 1.2 Прочитать спецификацию расширения (если есть)

Искать в `openspec/docs/` файл со спецификацией расширения. Извлечь:

- Префикс расширения (например, `pavIU_`).
- Точки перехвата (список `&Вместо`/`&Перед`/`&После`).
- Префикс журнала регистрации.

Если спецификация не найдена — попросить у пользователя префикс расширения.

### 1.3 Собрать список .bsl файлов

Использовать Glob по директории расширения (`src/*/cfe/<название>/`). Перечислить все `.bsl` файлы с путями.

### 1.4 Загрузить артефакты активного change (если есть)

Если ревью в контексте change:

```bash
openspec instructions apply --change "<name>" --json
```

Прочитать `design.md`, `tasks.md` — для понимания контрактов и ожидаемого поведения.

### 1.5 Сформировать контекст для ревьювера

Подготовить структурированный бриф. Использовать его в промптах батчей (шаг 2.1). Шаблон:

- **Расширение:** название, префикс, префикс журнала (если есть).
- **Файлы в scope:** по батчу — список .bsl с путями (из 1.6: группы A/B/C).
- **Стандарты:** 1c-coding-standards.mdc, 1c-vendor-standards/SKILL.md, конфигурация onec-code-reviewer.md.
- **Режим:** mode=prerelease; для каждого замечания указывать kind (functional / style).

### 1.6 Классификация модулей

Автоматически разбить список .bsl файлов на группы:

- **Группа A (глубокий ревью):** собственные модули расширения — путь содержит префикс расширения (например `рг` в имени папки/модуля: `ргСервисКриптографии*`, `ргЭкземпляры*` и т.д.). Содержат оригинальный код проекта.
- **Группа B (стандартный ревью):** перехваты типовых модулей — общие/объектные модули с аннотациями &Вместо/&Перед/&После/&ИзменениеИКонтроль, но не в группе A. Проверка: аннотации, корректность структуры #Вставка/#Удаление (директивы не удалять — это синтаксис переопределения), минимальные паттерны.
- **Группа C (лёгкий ревью):** модули форм — пути содержат `\Forms\` и `Form\Module.bsl` или `Form.xml`. Структура, обработчики, базовые паттерны.

Указать в контексте: количество файлов и суммарное число строк (если доступно) в каждой группе. Если группа A содержит >10 файлов — разбить её на 2 батча (A1 и A2 по 5–8 файлов), иначе ревьювер не осилит глубокий проход.

## Шаг 2. Батчевый ревью параллельными субагентами

### 2.0 Входные данные

Использовать группы A/B/C из шага 1.6 и общий контекст из 1.5. Имя расширения, префикс, стандарты, режим mode=prerelease — едины для всех батчей.

### 2.1 Параллельные вызовы onec-code-reviewer (до 3 одновременно)

Вызвать **Task** с `subagent_type: onec-code-reviewer` для каждого батча. Запускать батчи параллельно, где возможно.

**Батч 1 — Группа A (глубокий ревью):**
- Файлы: список модулей группы A (или A1, если A разбита).
- Промпт: «Провести **глубокий** предрелизный ревью собственных модулей расширения. Прочитай КАЖДЫЙ указанный файл целиком. Проверь: логику, потоки данных, обработку ошибок, Performance (запросы в циклах, индексы), Security (инъекции, доступ), БСП compliance, аннотации расширений, структуру модуля, документацию, Specific 1C Patterns (ТекущаяДата/ТекущаяДатаСеанса, Сообщить, Свойство, НСтр). Отдельно проверь **релизную чистоту комментариев**: маркеры в комментариях (автор, дата, заявка, строки // НАЧАЛО/КОНЕЦ, +++/---), временные инструкции, закомментированный старый код — такие замечания = HIGH [release-hygiene]. Директивы #Вставка/#Удаление не удалять и не помечать как мусор — это синтаксис переопределения расширения., обязательны до релиза. Проверь также: (1) дублирование #Область/#КонецОбласти, (2) опечатки и смешение кириллицы/латиницы в пользовательских строках, (3) заглушки (функции, возвращающие пустые/невалидные значения — проверяется всегда, не только в prerelease), (4) ЭтаФорма вместо ЭтотОбъект в оповещениях, (5) Попытка/Исключение без логирования при проглатывании ошибки, (6) Экспорт в секции СлужебныеПроцедурыИФункции, (7) имя метода противоречит директиве компиляции, (8) **транзакционные паттерны**: НачатьТранзакцию без Попытки/Исключения, НачатьТранзакцию без ЗафиксироватьТранзакцию/ОтменитьТранзакцию, отсутствие ОтменитьТранзакцию в Исключении, пользовательское взаимодействие внутри транзакции, (9) **утечка ресурсов**: COM-объекты без Попытки/Исключения с освобождением, HTTP/FTP-соединения без обработки ошибок, файловые операции без закрытия в пути ошибки. Для каждого замечания укажи файл:строка, уровень (с учётом prerelease-эскалации), kind (functional, style или release-hygiene), категорию, описание, рекомендацию. РЕЖИМ: mode=prerelease.»

**Батч 2 — Группа B (стандартный ревью):**
- Файлы: список модулей группы B.
- Промпт: «Провести предрелизный ревью перехватов типовых модулей. Фокус: аннотации &Вместо/&Перед/&После/&ИзменениеИКонтроль, корректность структуры #Вставка/#Удаление (директивы не удалять), префикс перехватов, Module Structure, Method Documentation, Code Cleanliness, Specific 1C Patterns. Отдельно проверь релизную чистоту комментариев (маркеры в комментариях, временные инструкции, закомментированный старый код) — HIGH [release-hygiene]. Директивы #Вставка/#Удаление не удалять. Проверь: дублирование #Область/#КонецОбласти, опечатки/кодировка в строках, заглушки (проверяется всегда, не только в prerelease), ЭтаФорма вместо ЭтотОбъект, Попытка/Исключение без логирования, Экспорт в СлужебныеПроцедурыИФункции, имя метода vs директива, **транзакционные паттерны**: НачатьТранзакцию без Попытки, без ЗафиксироватьТранзакцию/ОтменитьТранзакцию, пользовательское взаимодействие внутри транзакции. Для каждого замечания: файл:строка, уровень, kind (functional/style/release-hygiene), категория, описание, рекомендация. РЕЖИМ: mode=prerelease.»

**Батч 3 — Группа C (лёгкий ревью):**
- Файлы: список модулей группы C (модули форм).
- Промпт: «Провести предрелизный ревью модулей форм расширения. Фокус: структура, обработчики событий, аннотации перехватов в формах, базовые паттерны (ТекущаяДата, Сообщить, НСтр). Отдельно проверь релизную чистоту комментариев (маркеры в комментариях; не трогать #Вставка/#Удаление) — HIGH [release-hygiene]. Проверь: дублирование #Область/#КонецОбласти, опечатки/кодировка в пользовательских строках, ЭтаФорма вместо ЭтотОбъект, имя процедуры vs директива (&НаСервере/НаКлиенте). Для каждого замечания: файл:строка, уровень, kind (functional/style/release-hygiene), категория, описание, рекомендация. РЕЖИМ: mode=prerelease.»

Если группа A разбита на A1 и A2 — выполнить два отдельных вызова для A1 и A2 с тем же «глубоким» промптом; батчи B и C можно запускать параллельно с A1/A2.

### 2.2 Сведение отчётов

- Объединить выводы всех батчей в один список замечаний.
- Дедуплицировать по ключу «файл:строка:категория» (одно и то же место — одно замечание).
- Сохранить атрибут **kind** (functional / style / release-hygiene) у каждого замечания для шага 3 и 4.

## Шаг 3. Формирование предрелизного отчёта

На основе сводного списка замечаний (шаг 2.2) составить итоговый отчёт. Указывать **kind** (functional / style / release-hygiene) у каждого замечания; в сводке — разбивка по kind (functional и release-hygiene обязательны до релиза, style — по желанию).

```markdown
## Предрелизное ревью: [название расширения]

**Дата:** [дата]
**Scope:** N файлов (A: n, B: n, C: n), M процедур/функций
**Режим:** предрелизный (эскалация: LOW→MEDIUM, MEDIUM→HIGH для kind=functional; release-hygiene MEDIUM→HIGH)
**Стандарты:** 1c-coding-standards.mdc, вендорские (1c-vendor-standards, .cursor/docs/standard/ при необходимости), [спецификация расширения]

### Сводка

| Уровень  | functional | release-hygiene | style |
|----------|------------|----------------|-------|
| CRITICAL | N          | —              | —     |
| HIGH     | N          | N              | N     |
| MEDIUM   | N          | N              | N     |
| LOW      | N          | N              | N     |

(Эскалация для functional и release-hygiene; kind=style помечается тегом [style] и не эскалируется.)

### CRITICAL — блокеры релиза

- **`файл.bsl:строка`** — [Категория]  
  Проблема: описание.  
  Рекомендация: что сделать.

### HIGH — исправить до релиза

- **functional:** ...
- **release-hygiene:** ...
- **[style]:** ...

### MEDIUM — желательно исправить

- ...

### LOW — технический долг

- ...

### Итог

[Готов к релизу / Не готов к релизу]

Блокеры (CRITICAL): [список или «нет»]
Обязательны до релиза (HIGH functional): [список или «нет»]
Обязательны до релиза (HIGH release-hygiene): [список или «нет»]
HIGH style: [по желанию]
```

## Шаг 4. Предложение openspec change

Если обнаружены замечания уровня CRITICAL или HIGH:

Использовать **AskUserQuestion tool**:

> «Обнаружено N замечаний уровня CRITICAL/HIGH. Создать openspec change `prerelease-fix-<расширение>` для их устранения?»

**Варианты:**
- Да, создать change
- Нет, только отчёт

### Если пользователь выбрал «Да»

1. Создать новый change (имя в kebab-case, например `prerelease-fix-rg-kontur-cryptoapi`):

```bash
openspec new change "prerelease-fix-<расширение-kebab>" --schema prerelease-fix
```

Если схема `prerelease-fix` недоступна — использовать `--schema spec-driven` и создать обязательный артефакт specs по инструкции схемы.

2. Сформировать proposal (описание задачи) на основе отчёта:
   - Суть: устранение замечаний предрелизного ревью расширения `<название>`.
   - Список замечаний CRITICAL и HIGH (с указанием kind: functional / style).
   - Ссылка на отчёт: `reports/prerelease-review-ГГГГ-ММ-ДД.md`.

3. Сформировать tasks.md по правилам:
   - **CRITICAL** = отдельная задача на каждое замечание (блокер).
   - **HIGH functional** = отдельная задача на каждое замечание.
   - **HIGH style / MEDIUM (массовые):** если замечание одного типа затрагивает >3 файлов — одна задача с подзадачами по файлам (чекбоксы-подпункты). Пример:
     ```markdown
     - [ ] **2.2** [HIGH/style] Разметка #Область
       - [ ] CommonModules\РаботаСЭП\Ext\Module.bsl
       - [ ] CommonModules\РаботаСКартинками\Ext\Module.bsl
       - [ ] ...
     ```
   - Остальные MEDIUM — группировать по категории (1 задача на категорию).
   - **LOW** — одна групповая задача «Технический долг».

   Формат задачи (одиночное замечание):
   ```markdown
   - [ ] [CRITICAL] Файл:строка — Категория: Описание. Рекомендация.
   - [ ] [HIGH] [functional] Файл:строка — ...
   ```

4. Сохранить отчёт ревью в `openspec/changes/prerelease-fix-<расширение>/reports/prerelease-review-ГГГГ-ММ-ДД.md`.

5. Сообщить пользователю:
   - Название созданного change.
   - Количество задач в tasks.md.
   - Следующий шаг: `/opsx:apply prerelease-fix-<расширение>`.

## Граничные случаи

- **Замечаний нет**: сообщить «Замечаний, блокирующих релиз, не найдено. Расширение готово к релизу.»
- **Только LOW**: предложить создать задачу технического долга или пропустить.
- **Нет спецификации расширения**: выполнить ревью с общими правилами; в отчёте отметить, что проверка соответствия спецификации не выполнена.
- **Нет активного change**: ревью выполняется в standalone-режиме; отчёт сохранить в `temp/reports/prerelease-review-<расширение>-ГГГГ-ММ-ДД.md`.

## Интеграция

- Ревьювер: `onec-code-reviewer` v1.2 с поддержкой `mode=prerelease` и новыми категориями.
- Стандарты вендора и навигация: `AGENTS.md` → секция «Стандарты вендора 1С»; `.cursor/docs/standard/` (1c-standards-navigator.md, std-*.md) и `.cursor/skills/1c-vendor-standards/SKILL.md` — единый набор для предрелизной проверки.
- Если после ревью обнаружены архитектурные замечания (новые объекты, изменение API, структуры хранения) — не создавать задачи writer, остановиться и показать пользователю.
- Отчёт сохраняется согласно `preserve-subagent-reports.mdc`.
