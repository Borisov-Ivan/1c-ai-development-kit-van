## ЗАПРЕТ: прямое написание/редактирование .bsl файлов

**НИКОГДА** не пиши и не редактируй `.bsl` файлы напрямую.

**Обязательный порядок:**

1. Перед любой работой с BSL — загрузи скилл: `.cursor/skills/1c-extensions/SKILL.md` (аннотации) или `.cursor/skills/1c-feature-dev-enhanced/SKILL.md` (полный цикл).
2. При доработке (не точечная правка): план и ревью плана — через `onec-code-architect` (Task tool, `subagent_type="onec-code-architect"`); не переходить к реализации без плана и ревью/одобрения. Подробности: `.cursor/rules/1c-agent-delegation.mdc`.
2.1. **DELEGATION GATE:** до вызова агента — только артефакты + до 3 точечных обращений к .bsl. Разбор логики/вызовов/зависимостей — работа агента. Подробности: `.cursor/rules/1c-agent-delegation.mdc`, секция DELEGATION GATE.
3. Делегируй написание BSL-кода агенту `onec-code-writer` (Task tool, `subagent_type="onec-code-writer"`).
3.1. **Ограничение полномочий агентов-исполнителей:** разрешена только правка существующих .bsl файлов. Запрещено создание новых файлов/папок и изменение метаданных (.xml, Configuration.xml). Если нужного модуля или объекта нет в проекте — не добавлять самостоятельно: СТОП и сообщить пользователю, что именно и с какими параметрами добавить и выгрузить в проект для продолжения работы.
4. Обязательное ревью через `onec-code-reviewer` (Task tool, `subagent_type="onec-code-reviewer"`).
5. Critical/High **без архитектурных изменений** — авто-исправление через `onec-code-writer` (до 2 итераций), затем повторное ревью.
6. Critical/High **с архитектурными изменениями** (новые объекты, изменение API, потоков данных, структуры хранения, RLS) — СТОП, показать пользователю.
7. Только после PASS или явного одобрения пользователя — задача выполнена.

**Единственное исключение:** однострочные правки (опечатка, переименование переменной), изменения только в комментариях, или явный запрос пользователя «без агентов» / «без ревью».

**Если нарушил** — ОСТАНОВИСЬ, отмени правку, делегируй агенту.

Подробности: `.cursor/rules/1c-agent-delegation.mdc` (bsl-write-guard объединён)

## Анализ ошибок 1С (трассы и стеки)

**Приоритет над режимом:** Это правило действует **в любом режиме** (в т.ч. explore). Трассу не разбирать вручную. Если пользователь приложил файл трассы (`.pff`, `*_TRACE_*.txt`) или стек и просит разобрать — **перед** вызовом onec-trace-analyst **обязательно подготовить обогащённый контекст** (структурированный бриф), затем делегировать **onec-trace-analyst** с путём к трассе и брифом. Подробности — в `.cursor/rules/1c-error-analysis.mdc` (разделы ПОДГОТОВКА КОНТЕКСТА и ДЕРЕВО РЕШЕНИЙ). В режиме explore делегирование trace-analyst / explorer / architect не отменяется; ручной разбор трассы не подменяет вызов агента.

**Когда:** ошибка 1С + трасса/стек (файл `.pff`, `*_TRACE_*.txt`), команда /debug, запросы «проанализируй трассу», «trace analysis».

**Фокус анализа задаётся запросом пользователя** — не захардкоженными паттернами (не «ищи только записи/транзакции»). В trace-analyst передавать контекст: вопрос пользователя или описание ошибки.

- **onec-trace-analyst** (Task tool, `subagent_type="onec-trace-analyst"`): путь к трассе + контекст запроса; проверка контекста по коду (Read исходников для критичных строк); при недостатке данных в COMPACT — явно требовать TRACE_FULL, не додумывать.
- **onec-code-explorer:** задача формируется по отчёту trace-analyst — ключевые точки/находки из отчёта, а не фиксированный шаблон (например «все точки записи объекта X»).
- **onec-code-architect:** вызывать, когда trace-analyst или explorer указал архитектурную проблему (любую из отчёта).

Полное дерево решений, пороги делегирования и готовые формулировки для субагентов — в `.cursor/rules/1c-error-analysis.mdc`.

## Предрелизное ревью расширения 1С

**Команда:** `/prerelease-review` или скилл `.cursor/skills/prerelease-review/SKILL.md`.

**Когда использовать:** перед выпуском релиза расширения 1С — финальная проверка качества кода.

**Что делает:**
1. Собирает scope (файлы расширения), стандарты и спецификацию расширения.
2. Вызывает `onec-code-reviewer` с `mode=prerelease` — активирует эскалацию уровней.
3. Формирует отчёт: CRITICAL / HIGH / MEDIUM / LOW, файл:строка, рекомендации, итог (готов/не готов).
4. Предлагает создать openspec change на устранение найденных замечаний.

**Эскалация уровней в предрелизном режиме:**
- LOW → MEDIUM: чейнджлог-маркеры, ссылки на артефакты дизайна в комментариях, отсутствие заголовка модуля.
- MEDIUM → HIGH: отсутствие `#Область` разметки, нет документации экспортных методов, мёртвый код, дублирование логики.

**Новые категории ревьювера (работают в любом режиме):** аннотации расширений (&Перед/&После/&Вместо/&ИзменениеИКонтроль), структура модуля (#Область), документация методов, именование в расширениях (префикс), чистота кода, паттерны 1С (ТекущаяДата, Сообщить, ?(), НСтр).

## Полнота артефакта specs (Delta Specs Gate)

**Приоритет над режимом:** действует в любом режиме (explore, apply, verify, archive).

При работе с change в схеме spec-driven агент проверяет наличие delta specs (`openspec/changes/<name>/specs/`). Если specs отсутствуют:

- **explore:** предложить создать в конце сессии (не блокировать).
- **apply:** WARNING перед началом реализации с предложением создать.
- **verify:** CRITICAL в отчёте Completeness.
- **archive:** предупреждение с предложением создать (как для незавершённых задач).

Подробности: `.cursor/rules/openspec-specs-gate.mdc`.
